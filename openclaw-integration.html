Get Started

One skill. One server. One command. That’s it.

1

Install the Skill

This is the entire integration. One folder, two files - a markdown doc and a shell helper. Drop it in and OpenClaw learns hardware.

install - skill 
cp -r skill/wireclaw ~/.openclaw/workspace/skills/wireclaw

Or ask OpenClaw: “Install the wireclaw skill from github.com/M64GitHub/WireClaw - it’s in the skill/wireclaw folder.”

2

NATS Server + CLI

The bridge between OpenClaw and WireClaw. Ask OpenClaw: “Install nats-server and the nats CLI, then start the server.”

Or do it yourself:

install - NATS 
# macOS
brew install nats-server nats-io/nats-tools/nats
nats-server -a 0.0.0.0 &

# Linux
curl -sf https://install.nats.dev | sh
nats-server -a 0.0.0.0 &

Point WireClaw at it: set nats_host in the web config portal. Non-default URL? export WIRECLAW_NATS_URL="nats://192.168.1.100:4222"

3

Talk to Hardware

What You Can Build

Concrete ideas. Most take one or two commands.

CI Build Light

OpenClaw watches your GitHub Actions. Green LED on pass, red on fail. Push once - WireClaw handles the rest with a nats_value sensor and a persistent rule.

Morning Wake-Up Light

Warm orange LED at 7:30 AM, off at 8:00. One time-based rule on clock_hhmm. No cron, no cloud. The chip just knows.

Weather-Reactive Desk Lamp

OpenClaw checks the forecast, publishes temperature to NATS. WireClaw’s nats_value sensor picks it up - blue when cold, red when hot, green when perfect. Autonomous after setup.

Temperature Telegram Alert

NTC sensor on an ESP32 near your server rack. Rule fires a Telegram message when it crosses 40°C. No polling. No lambda. Just a rule on a $5 chip.

Live Event Monitor

Subscribe to wireclaw-01.events and stream rule triggers, sensor readings, and status changes in real time. Debug rules, log data, or feed into dashboards - all through NATS.

How It Works

Flat JSON over NATS request/reply. OpenClaw’s LLM reads the skill, generates the payloads. WireClaw executes on hardware.

nats - request/reply 
$ nats req wireclaw-01.tool_exec '{"tool":"led_set","r":128,"g":0,"b":255}'
{"ok":true,"result":"LED set to RGB(128, 0, 255)"}

$ nats req wireclaw-01.tool_exec '{"tool":"sensor_read","name":"chip_temp"}'
{"ok":true,"result":"chip_temp: 33.2 C"}

NATS Subjects

{device}.tool_execExecute any tool, get result. Request/reply.
{device}.capabilitiesQuery available tools, devices, rules.
{device}.eventsSubscribe to rule triggers, sensor readings, and status changes. Stream with nats sub wireclaw-01.events.
_wc.discoverFind all WireClaws on the network. Multiple responses.

Cross-Domain Patterns

Digital triggers physical. Physical triggers digital. Data bridges both.

Digital → Physical

pattern

OpenClaw detects something. WireClaw reacts in hardware.

OpenClaw
When my GitHub CI fails, flash the desk LED red
Watching CI status. On failure, sending led_set(255,0,0) to wireclaw-01.
nats - digital to physical 
$ nats req wireclaw-01.tool_exec '{"tool":"led_set","r":255,"g":0,"b":0}'
{"ok":true,"result":"LED set to RGB(255, 0, 0)"}
flow
CI fail → OpenClaw → NATS → WireClaw → LED red

Physical → Digital

pattern

WireClaw rule fires. OpenClaw subscribes and reacts digitally.

nats - physical to digital 
# WireClaw: alert when overheating
$ nats req wireclaw-01.tool_exec '{
    "tool":"rule_create",
    "rule_name":"temp alert",
    "sensor_name":"chip_temp",
    "condition":"gt","threshold":35,
    "on_action":"nats_publish",
    "on_nats_subject":"alerts.overheating",
    "on_nats_payload":"Temperature: {value}C"
  }'
{"ok":true,"result":"Rule created: temp alert"}

# OpenClaw: subscribe and react
$ nats sub alerts.overheating
flow
sensor → rule fires → NATS → OpenClaw → reacts digitally

Data Bridge

most powerful

OpenClaw pushes data to NATS. WireClaw has a virtual sensor on that subject. Persistent rules handle the rest.

nats - data bridge 
# Register a virtual sensor fed by NATS
$ nats req wireclaw-01.tool_exec '{
    "tool":"device_register",
    "name":"ci_status","type":"nats_value",
    "subject":"ci.build.status"
  }'
{"ok":true,"result":"Registered: ci_status (nats_value)"}

# Create rule: CI fail → red, CI pass → green
$ nats req wireclaw-01.tool_exec '{
    "tool":"rule_create",
    "rule_name":"ci alert","sensor_name":"ci_status",
    "condition":"eq","threshold":0,
    "on_action":"led_set","on_r":255,"on_g":0,"on_b":0,
    "off_action":"led_set","off_r":0,"off_g":255,"off_b":0
  }'
{"ok":true,"result":"Rule created: ci alert"}

# OpenClaw just pushes status - WireClaw handles everything
$ nats pub ci.build.status "0"   fail → red
$ nats pub ci.build.status "1"   pass → green
data flow
OpenClaw pushes → NATS → virtual sensor → rule engine → LED
✓ After setup, WireClaw runs the rules autonomously. OpenClaw just pushes data. No polling. No LLM in the loop. The chip handles everything locally.

Inside The Skill

One folder. Two files. That’s what teaches an AI agent to control hardware.

MD

SKILL.md

Full tool reference - every parameter, every constraint, every edge case. Derived from WireClaw’s actual system prompt. OpenClaw generates correct tool calls on the first try.

SKILL.md - what OpenClaw sees 
---
name: wireclaw
description: >
  Control WireClaw IoT devices on your network.
  Use when the user wants to interact with physical
  hardware: LEDs, GPIO pins, sensors, relays ...
tools:
  - Bash
  - Read
metadata:
  openclaw:
    requires:
      binaries:
        - nats
    env:
      - WIRECLAW_NATS_URL
---

# WireClaw - Physical World Automation for OpenClaw
# ... 19 tools, examples, constraints, patterns

It’s just markdown. OpenClaw reads it once, learns every tool, and generates correct NATS payloads. No plugins. No SDK. No config.

SH

wc.sh

Convenience wrapper for NATS calls:

wc.sh - usage 
wc.sh exec <device> <tool> [params]   # Execute a tool
wc.sh caps <device>                    # Query capabilities
wc.sh discover                         # Find all devices
wc.sh sub  <device>                    # Subscribe to events

Security Model

What’s enforced, what’s blocked, and how to lock it down.

Blocked Tools

remote_chatUses re-entrant NATS processing that corrupts state inside a callback. Use nats req other-device.tool_exec instead.
file_write to /memory.txtInternal AI memory - external writes would corrupt the device’s learned context.

Network Security

v1 has no built-in authentication - anyone on the NATS network can call tools. For trusted home networks, this is fine. For anything else:

nats-server.conf - lock it down 
# Require credentials
authorization {
  users = [
    { user: "openclaw", password: "...",
      permissions: { publish: ">", subscribe: ">" } }
    { user: "wireclaw", password: "...",
      permissions: { subscribe: ["wireclaw-01.>", "_wc.>"],
                     publish: ">" } }
  ]
}

# Bind to localhost only (if same machine)
listen: 127.0.0.1:4222

NATS supports user/password, token, TLS, and NKey authentication. Subject-level permissions let you restrict exactly which devices each client can talk to.
